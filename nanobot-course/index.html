Module 1

Meet nanobot: one message's journey

Imagine you text your nanobot on Telegram: "remind me to call mom at 5pm, and check what the weather is like first." Less than two seconds later you are watching it type back. This module is everything that happens in between.

First, the plain version of what nanobot even is: it is an open-source, self-hosted personal AI assistant. You run it on your own machine or server, it connects to chat apps you already use — Telegram, Discord, Slack, WeChat, a built-in web chat window, and more — and it talks to a large language model (Claude, GPT, and others) as its "brain." It can also use tools to actually get things done, not just chat: reading and writing files, running commands, searching the web. The project's own tagline is blunt about it: an open-source, ultra-lightweight personal AI agent you can truly own.

Why should you care how the message travels? Once you can picture this whole round trip, you can tell an AI coding tool exactly where to add a feature — "that belongs in a channel, not in the core loop" — instead of guessing. And when something breaks, you already know which of the handful of stops to suspect first.

Here is a way to hold the whole trip in your head: sending a message to nanobot is like dropping a letter in a mailbox. It travels to a sorting facility, gets picked up by a caseworker who reads it, may send runners out to fetch things — a weather report, a calendar lookup — or write something on your behalf, and then a reply comes back through the same path. This module walks that whole route once, at a human pace. Every later module will zoom into one single stop along it.

Notice that "MessageBus" shows up twice — once as an in-tray, once as an out-tray. That is not two different components, it is one shared post office with two windows. Keep scrolling and we will look at the actual code behind each stop.

The entire post office — the in-tray and out-tray from the animation above — is remarkably small. Here is essentially all of it.

nanobot/bus/queue.py:16-34 
def __init__(self):
    self.inbound: asyncio.Queue[InboundMessage] = asyncio.Queue()
    self.outbound: asyncio.Queue[OutboundMessage] = asyncio.Queue()

async def publish_inbound(self, msg: InboundMessage) -> None:
    """Publish a message from a channel to the agent."""
    await self.inbound.put(msg)

async def consume_inbound(self) -> InboundMessage:
    """Consume the next inbound message (blocks until available)."""
    return await self.inbound.get()

async def publish_outbound(self, msg: OutboundMessage) -> None:
    """Publish a response from the agent to channels."""
    await self.outbound.put(msg)

In plain English

This whole class is really just two lines that matter: an in-tray (inbound) and an out-tray (outbound). Both are a queueA list where things get added at one end and taken off the other, in order — like a single-file line at a mailroom window. First come, first served.: whichever message arrived first gets picked up first.

Every chat platform — Telegram, Slack, the web UI, all of them — drops its letters into that exact same in-tray. Nothing about the sorting facility needs to know which mailbox any given letter came from.

Aha!

Two independent queues, one for "in" and one for "out," is a classic pattern called a message bus. It lets two halves of a system — here, the chat channels and the agent's brain — talk to each other without either one ever needing to know how the other works internally. That is also exactly why nanobot can support a dozen chat platforms and several different AI providers without its core code caring which ones happen to be in use.

So who is actually standing at that in-tray window, watching for new letters? That is AgentLoop, and its main job is a loop that never really stops.

In plain English: it waits up to one second for a new letter to arrive. If nothing shows up in that second, it does not just sit idle — it uses the moment to do housekeeping, checking whether any conversation has gone quiet long enough that its memory should be tidied up, then goes right back to waiting.

That "wait, but do something useful instead of standing frozen" trick is possible because the whole loop is written as an async taskDoing something and moving on to other work while it finishes, instead of standing still and waiting — like AgentLoop starting an errand for your message and immediately going back to watch for the next one.. Nothing here demands your CPU sit and twiddle its thumbs for a whole second doing nothing.

Once AgentLoop actually has your letter in hand, it does something important: it does not sit there answering you before it is allowed to glance at anyone else's mail.

It spins your message off as its own independent errand (that async task again) and immediately goes back to watching the in-tray for the next letter. Your conversation and someone else's conversation on a different chat can genuinely be in progress at the same time. (Module 3 covers exactly how nanobot still keeps your own conversation in the right order while doing this.)

Somewhere inside that errand, AgentRunner is the one actually talking to the LLM and, if the model asks for it, dispatching "runners" out for tools — checking the weather, writing your reminder down. That is a whole module on its own later; for now, just picture it as the desk where your letter gets read and answered. Once it has an answer, the last step is simple.

nanobot/agent/loop.py:1087-1088 
if response is not None:
    await self.bus.publish_outbound(response)

Same trick, in reverse: the reply goes into the shared out-tray, and whichever channel adapter is watching for mail addressed to your Telegram chat picks it up and delivers it. That is the last stop in the animation above.

Quick check before we move on.

When your message arrives, does nanobot handle it immediately, or does it get in line?

The key thing to carry forward: nanobot's core is deliberately small, and a message is handed from stop to stop like a letter changing hands. Every stop only knows how to do its own one job — the in-tray does not need to know what a caseworker does, and the caseworker does not need to know how Telegram formats its messages. That separation of concerns is exactly why nanobot can support a dozen chat platforms and several AI providers without the core ever needing to change.

Module 2

Meet the cast

We just watched a letter travel the whole route. Now let's meet everyone working at each stop — the way a theater program lists its cast and crew before the curtain goes up, so you know exactly who's responsible for what before the show starts.

Knowing which actor owns which job is exactly what lets you tell an AI coding tool "add this feature as a new channel" or "that belongs in the prop table, not the director's script" — and to guess correctly which file to open when something breaks. Here's the full cast:

Channels

The ushers who speak the audience's language — each one translates a different platform's way of sending a message into one common form everyone backstage understands.

MessageBus

The stage manager's call sheet — the shared list everyone reads from and writes to, so the ushers never need to talk to the actors directly.

AgentLoop

The director — owns the whole show for one conversation: decides what happens next, in what order, and calls the cues.

AgentRunner

The lead actor actually improvising the scene — on stage doing the back-and-forth with the LLM and requesting props (tools) when needed.

ToolRegistry

The prop table — everything the actor is allowed to reach for, catalogued and ready to hand over on request.

Providers

The scriptwriter — the LLM actually generating the lines; nanobot can swap which scriptwriter it's using without changing anyone else's job.

Start with the ushers. Telegram, Discord, Slack, the web UI, and a dozen-plus others all speak differently under the hood — different APIs, different message formats. Every one of them is required to sign the same contractA rulebook every implementation must follow, even though each one fills in the details differently. In code this is often called an "abstract class" — a template that says "you must have a method called X," without saying how X works. before it's allowed backstage.

This is the contract every usher must sign. However Telegram's API and Slack's API differ under the hood, both must know how to start listening, stop, and hand a reply to their platform. Nobody else in the codebase needs to know those differences exist.

Add a new file that follows the contract from the last screen, and it's automatically found and put to work.

Once a message is in, it doesn't go straight to an actor. It gets dropped on the stage manager's call sheet — the MessageBus — and the director picks it up from there. That indirection is the whole point: the usher never needs to know or care which director is on duty, or what they're doing right now.

Once the director hands a scene to the lead actor, that actor sometimes needs a prop — read a file, run a shell command, search the web. It doesn't wander off backstage looking for one. It requests a prop by name from the prop table, and the prop table hands it over — or refuses, if that prop was never catalogued.

So who decides the director gets a lead actor to direct in the first place? One line, buried in setup:

And the lead actor's lines don't come from nowhere — they come from the scriptwriter. nanobot supports several: Claude, GPT, and others, all hidden behind one shared shape.

Module 3

Anatomy of a turn

Module 2 introduced the director and lead actor. Now watch them run one full scene, station by station — because when a conversation feels "stuck," or a reply mysteriously "forgets" something, knowing the fixed sequence of stations lets you point an AI tool at the one station that's misbehaving instead of "just look at the whole agent."

Think of one turn — one message in, one reply out — as a factory assembly line, or a relay race with fixed stations. The work doesn't happen in one big step. It moves forward, station by station, and each station does exactly one job before handing the (increasingly-finished) work to the next runner. Nobody skips a station out of order.

nanobot names every station in the line, right in one place — nothing is hidden elsewhere in the code:

nanobot/agent/loop.py:90-98

    RESTORE = auto()
    COMPACT = auto()
    COMMAND = auto()
    BUILD = auto()
    RUN = auto()
    SAVE = auto()
    RESPOND = auto()
    DONE = auto()

That's the whole route, top to bottom: RESTORE → COMPACT → COMMAND → BUILD → RUN → SAVE → RESPOND → DONE.

A list of station names isn't enough on its own — something has to decide what comes next. nanobot keeps that decision in one small rulebook, a lookup table of "if you're here, and this happened, go there":

nanobot/agent/loop.py:183-192 
_TRANSITIONS: dict[tuple[TurnState, str], TurnState] = {
    (TurnState.RESTORE, "ok"): TurnState.COMPACT,
    (TurnState.COMPACT, "ok"): TurnState.COMMAND,
    (TurnState.COMMAND, "dispatch"): TurnState.BUILD,
    (TurnState.COMMAND, "shortcut"): TurnState.DONE,
    (TurnState.BUILD, "ok"): TurnState.RUN,
    (TurnState.RUN, "ok"): TurnState.SAVE,
    (TurnState.SAVE, "ok"): TurnState.RESPOND,
    (TurnState.RESPOND, "ok"): TurnState.DONE,
}

Each station finishes and reports back one word — almost always "ok", meaning "done, keep going." This setup is called a state machineA system that is always in exactly one named state (station), with explicit rules for what state comes next. Instead of a tangle of if/else scattered everywhere, you get one small table you can read start to finish..

Look closely at COMMAND: it has two possible exits. Type a slash command like /new, and it takes the "shortcut" lane straight to DONE — skipping BUILD, RUN, and SAVE entirely. There's no need to ask the LLM anything for a command nanobot already knows how to handle itself.

What does each station actually do? Here's the whole relay, translated:

RESTORE

Recover any in-progress work left over from before a crash or interruption, and pull out any attached files or images.

COMPACT

Check whether this conversation's history has grown too long and needs summarizing first, before doing anything else.

COMMAND

Check whether the message is a slash command like /new or /stop. If so, handle it directly and take the shortcut exit.

BUILD

Assemble everything the LLM needs to see: the conversation history, the new message, and the current list of tools it's allowed to use.

RUN

The actual back-and-forth with the LLM and its tools — this is where Module 2's lead actor, AgentRunner, takes over.

SAVE

Write the finished turn to the conversation's permanent history, so it's there next time.

RESPOND

Package the final reply and send it back down the pipe to the channel — Telegram, Slack, wherever it came from.

DONE

The finish line. This turn is fully closed out; nanobot waits for whatever message arrives next.

The line runs one station at a time for a given conversation — but different conversations don't wait on each other at all. Each one gets its own turnstile:

In plain English

Every conversation gets its own lockA rule that only one thing at a time can hold — like a single key to a single door. Whoever has the key goes through; everyone else waits their turn for that same door. — a single-occupancy turnstile. Your conversation with nanobot can't have two turns processing at once and tripping over each other, but someone else's DM, or a different chat entirely, has its own separate turnstile and runs fully in parallel.

So what happens if you send a follow-up message while nanobot is still mid-reply to your first one? It doesn't start a second, competing turn:

Module 4

The outside world

Module 3 watched one turn run start to finish, entirely inside nanobot's own building. This module is about the border around that building — the interpreters waiting at the gate, the backup interpreter who steps in when the first one calls in sick, and the one checkpoint that every outbound request has to clear before it's allowed to leave.

Think of nanobot as a building with a border crossing. People arrive speaking different languages — Telegram's dialect, Slack's dialect, Discord's — and interpreters at the gate translate every one of them into a single shared form before anything moves further inside. If the building's usual translator for talking to the AI model is unavailable, a backup translator quietly takes over, so nobody in the building even notices the swap. And anything trying to leave the building — a tool reaching out to fetch a web page, or call another server — has to pass a checkpoint that refuses to let it wander into restricted territory.

Why this matters: nanobot runs with real power — it can read your files, run shell commands, fetch URLs. This module is what lets that power exist without becoming a security hole. It's also the part of the codebase you must never let an AI coding tool "quick fix" by routing a network call around it.

You already met the interpreters at the gate in Module 2: every channelA connector to one chat platform — Telegram, Slack, Discord, and so on. implements the same start / stop / send contract, so a Telegram message and a Slack message both arrive further inside as the exact same shape, an InboundMessage. That's the translation happening right at the door.

There's a second kind of interpreter, further in: the one that talks to the AI model itself. If you've configured backup models, nanobot doesn't wrap just one provider — it wraps a primary provider plus a list of fallbacks, ready to step in.

nanobot/providers/factory.py:184-197

In plain English

If a primary AI provider is down or rate-limited, this wrapper quietly tries the next one on the list instead. The rest of the codebase just calls "the provider" and never has to know whether it's talking to the first choice or the third — the swap is invisible from inside the building.

Now the checkpoint at the outbound side of the border. Before nanobot lets any tool fetch a URL, it checks the request against a no-fly list of address ranges that mean "somewhere private," never "the public internet."

Notice this checks the resolved address, not just the text of the URL. A sneaky domain name that looks public but is secretly pointed at an internal address still gets caught, because nanobot looks up the real destination first and checks that.

This whole family of trick — getting a server to fetch an address on your behalf that it shouldn't be able to reach — has a name: SSRFServer-Side Request Forgery: tricking a server into fetching an address it shouldn't be able to reach on your behalf — like tricking a hotel concierge into handing you a master key by asking very confidently.. It's one of the most common ways a "helpful" AI tool with network access gets turned into a break-in method.

Aha!

Notice this check lives in exactly one function that every tool calls, instead of being copy-pasted into every tool that touches the network. A security rule that exists in twenty places is a security rule that's wrong in at least one of them — a boundary is only as trustworthy as the single place everything is forced to go through.

When the checkpoint blocks something, nanobot doesn't just quietly fail — it tells the AI model exactly why, in words deliberately blunt enough to stop a "creative" model from trying to route around it.

Module 5

The clever tricks: sleeping on it

Module 4 was about nanobot's border with the outside world. This module is about what it does with everything it learns after it comes back inside.

You don't file away a day's experience as it happens. Your brain replays it later — while you sleep — turning a messy stream of moments into a few things actually worth keeping. nanobot does the same thing, on a schedule, and it's not a metaphor stretch: the codebase's own name for this mechanism is literally "Dream."

It keeps a running, append-only diary of everything that happens (history.jsonl), and periodically runs a separate Dream pass — a dedicated LLM callA single request sent to an AI model to get one response back — here, its only job is summarizing, not chatting. whose only job is to read the unprocessed diary entries and boil them down into durable notes (MEMORY.md, SOUL.md, USER.md).

Why should you care? Understanding that memory consolidation happens separately from live conversation explains why nanobot can stay useful over weeks without its context window overflowing — and it's the same reason you can ask it to "keep working on this for a while" and trust it won't lose the thread.

First, the diary itself. Every time something happens — a message, a tool call, a result — nanobot appends one more line to history.jsonl. Never edited, never reordered, only ever added to. Each entry gets the next ticket number in line, and handing out that number has to be atomicEither the whole thing happens, or none of it does — never caught halfway., or two things writing at the same instant could grab the same number and quietly clobber each other.

The with self._append_lock: line is what makes it atomic — from the outside, "grab the next number and write the entry" looks like one instant, uninterruptible step, not two steps that something else could squeeze between.

Diaries occasionally need rewriting wholesale — say, trimming old entries once they've been dreamed about. That's the riskiest kind of file operation there is: if the process dies halfway through, you can be left with a half-written, corrupted file where your entire history used to be. Here's how nanobot avoids that.

nanobot/agent/memory.py:446-455 
def _write_entries(self, entries: list[dict[str, Any]]) -> None:
    """Overwrite history.jsonl with the given entries (atomic write)."""
    tmp_path = self.history_file.with_suffix(self.history_file.suffix + ".tmp")
    try:
        with open(tmp_path, "w", encoding="utf-8") as f:
            for entry in entries:
                f.write(json.dumps(entry, ensure_ascii=False) + "\n")
            f.flush()
            os.fsync(f.fileno())
        os.replace(tmp_path, self.history_file)

In plain English

Instead of erasing the real diary and writing the new version into the same spot, nanobot writes the entire new version to a scratch file first, makes sure it's fully saved to disk (fsync), and only then swaps the scratch file in to replace the real one in one atomic move (os.replace). There is never a moment where the real file is partially written — it's either the complete old version, or the complete new one.

Aha!

Notice the crash-safety trick here is dumb, boring, and 100% reliable — no AI involved. The judgment call (what's worth remembering) is the one part handed to an LLM. That split — rigid code for the parts that must never fail, a smarter model for the parts that need judgment — shows up everywhere in well-built AI systems.

So how does the Dream pass know what's new since it last ran? It remembers the ticket number of the last diary entry it already processed, and only feeds the entries after that into the next pass.

The big picture: the cockpit and the flight recorder

Module 5 was about what nanobot remembers. This last module is about the app you're probably reading this on, and what to do when something goes sideways.

Picture a cockpit. Every dial, switch, and screen the pilot touches is just a different way of reaching the same aircraft underneath — and behind all of it sits a flight recorder, quietly timestamping everything that happens, so if something ever goes wrong, nobody has to guess what happened. That's this module: the web app you might be looking at right now is just another dial on the same aircraft as Telegram and Slack, and nanobot keeps its own flight recorder for exactly the moments when you need to know what went wrong.

This is the payoff for the whole course. Once you can see the cockpit's layout, "I understand how this works" turns into "I know exactly what to ask an AI coding tool to check first" when a reply is slow or wrong — and where to add a new feature so it doesn't fight the grain of the design.

Here's the reveal: the WebUI — the browser tab serving this very page — is not a special case wired directly into nanobot's core. It's just another implementation of the same WebSocketA phone line that stays open between your browser and the server, so either side can talk at any moment, instead of the browser having to call back and ask "anything new?" over and over. channel contract from Module 2 — the same start / stop / send shape every channel follows, whether it's Telegram, Slack, or your own browser tab. It just happens to also serve up the compiled WebUI's HTML and JS files.

No shortcuts, no side door into the agent core. The cast member you're talking to right now walked in through the same stage door as everyone else.

So how does your browser actually talk to nanobot over that open phone line? In small, labeled packages. Every message that crosses the WebSocket — a new chat message, a streamed reply chunk, a status update — is a JSON object with a type field saying exactly what kind of thing it is.

nanobot/channels/websocket.py:199-218 
def _parse_envelope(raw: str) -> dict[str, Any] | None:
    """Return a typed envelope dict if the frame is a new-style JSON envelope, else None."""
    text = raw.strip()
    if not text.startswith("{"):
        return None
    try:
        data = json.loads(text)
    except json.JSONDecodeError:
        return None
    if not isinstance(data, dict):
        return None
    t = data.get("type")
    if not isinstance(t, str):
        return None
    return data

In plain English

Every message flowing over the WebSocket connection between your browser tab and nanobot is a small labeled package — a JSON object with a type field. The browser and the server both speak this same small, explicit vocabulary. Nothing is inferred or guessed from raw text; if it doesn't match the shape, it's simply not treated as a valid envelope.

Aha!

This is why well-built systems use explicit, typed protocols instead of parsing free-form text: a labeled envelope can be validated and rejected safely if it's malformed, while "guess what the sender meant" code invites silent bugs and security holes. Explicit beats implicit whenever two systems need to trust each other.
