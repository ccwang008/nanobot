Module 3

Anatomy of a turn

Module 2 introduced the director and lead actor. Now watch them run one full scene, station by station — because when a conversation feels "stuck," or a reply mysteriously "forgets" something, knowing the fixed sequence of stations lets you point an AI tool at the one station that's misbehaving instead of "just look at the whole agent."

Think of one turn — one message in, one reply out — as a factory assembly line, or a relay race with fixed stations. The work doesn't happen in one big step. It moves forward, station by station, and each station does exactly one job before handing the (increasingly-finished) work to the next runner. Nobody skips a station out of order.

nanobot names every station in the line, right in one place — nothing is hidden elsewhere in the code:

nanobot/agent/loop.py:90-98 
class TurnState(Enum):
    RESTORE = auto()
    COMPACT = auto()
    COMMAND = auto()
    BUILD = auto()
    RUN = auto()
    SAVE = auto()
    RESPOND = auto()
    DONE = auto()

That's the whole route, top to bottom: RESTORE → COMPACT → COMMAND → BUILD → RUN → SAVE → RESPOND → DONE.

A list of station names isn't enough on its own — something has to decide what comes next. nanobot keeps that decision in one small rulebook, a lookup table of "if you're here, and this happened, go there":

nanobot/agent/loop.py:183-192 
_TRANSITIONS: dict[tuple[TurnState, str], TurnState] = {
    (TurnState.RESTORE, "ok"): TurnState.COMPACT,
    (TurnState.COMPACT, "ok"): TurnState.COMMAND,
    (TurnState.COMMAND, "dispatch"): TurnState.BUILD,
    (TurnState.COMMAND, "shortcut"): TurnState.DONE,
    (TurnState.BUILD, "ok"): TurnState.RUN,
    (TurnState.RUN, "ok"): TurnState.SAVE,
    (TurnState.SAVE, "ok"): TurnState.RESPOND,
    (TurnState.RESPOND, "ok"): TurnState.DONE,
}

Each station finishes and reports back one word — almost always "ok", meaning "done, keep going." This setup is called a state machineA system that is always in exactly one named state (station), with explicit rules for what state comes next. Instead of a tangle of if/else scattered everywhere, you get one small table you can read start to finish..

Look closely at COMMAND: it has two possible exits. Type a slash command like /new, and it takes the "shortcut" lane straight to DONE — skipping BUILD, RUN, and SAVE entirely. There's no need to ask the LLM anything for a command nanobot already knows how to handle itself.

What does each station actually do? Here's the whole relay, translated:

RESTORE

Recover any in-progress work left over from before a crash or interruption, and pull out any attached files or images.

COMPACT

Check whether this conversation's history has grown too long and needs summarizing first, before doing anything else.

COMMAND

Check whether the message is a slash command like /new or /stop. If so, handle it directly and take the shortcut exit.

BUILD

Assemble everything the LLM needs to see: the conversation history, the new message, and the current list of tools it's allowed to use.

RUN

The actual back-and-forth with the LLM and its tools — this is where Module 2's lead actor, AgentRunner, takes over.

SAVE

Write the finished turn to the conversation's permanent history, so it's there next time.

RESPOND

Package the final reply and send it back down the pipe to the channel — Telegram, Slack, wherever it came from.

DONE

The finish line. This turn is fully closed out; nanobot waits for whatever message arrives next.

The line runs one station at a time for a given conversation — but different conversations don't wait on each other at all. Each one gets its own turnstile:

nanobot/agent/loop.py:1033-1042 
lock = self._session_locks.setdefault(session_key, asyncio.Lock())
gate = self._concurrency_gate or nullcontext()

pending: asyncio.Queue | None = None
try:
    async with lock, gate:
        # Only the task that owns the session lock may publish the
        # active mid-turn injection queue for this session.
        pending = asyncio.Queue(maxsize=20)
        self._pending_queues[session_key] = pending

In plain English

Every conversation gets its own lockA rule that only one thing at a time can hold — like a single key to a single door. Whoever has the key goes through; everyone else waits their turn for that same door. — a single-occupancy turnstile. Your conversation with nanobot can't have two turns processing at once and tripping over each other, but someone else's DM, or a different chat entirely, has its own separate turnstile and runs fully in parallel.

So what happens if you send a follow-up message while nanobot is still mid-reply to your first one? It doesn't start a second, competing turn:

Your second message gets slipped into a small waiting line for that same conversation's current turn to notice and fold in — rather than nanobot trying to answer both at once and potentially replying twice, out of order.

Inside the RUN station specifically, there's another shortcut worth knowing: tool calls that don't depend on each other don't wait in line either.

nanobot/agent/runner.py:1138-1145 
if spec.concurrent_tools and len(batch) > 1:
    batch_results = await asyncio.gather(*(
        self._run_tool(
            spec, tool_call, external_lookup_counts, workspace_violation_counts,

In plain English

If the LLM asks for three things at once — say, check the weather, read a file, and search the web — none of those three need to wait on each other. nanobot fires them all off together and waits for all three to finish, instead of doing them one at a time.

Aha!

Any time a process has "phases" — a checkout flow, a deploy pipeline, an onboarding wizard — a state machine like this one is often how it's actually built under the hood. It's also exactly why good logs name the phase, not just "something broke": a table of named stations turns "the agent is misbehaving" into "RUN failed on this turn," a question an AI tool can actually go answer.
